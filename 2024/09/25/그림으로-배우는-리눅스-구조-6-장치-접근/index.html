<그림으로 배우는 리눅스 구조> 6. 장치 접근

2024-09-25

책을 읽으면서 중요한 내용을 기록해 두려고 한다.

프로세스는 장치에 어떻게 접근할까?

프로세스는 장치에 직접 접근하지 못한다. 왜일까? 여러 프로그램이 동시에 장치에 접근해 문제가 발생 할 수 있고 다른 중요 데이터가 훼손될 수 있기 때문이다. 그래서 프로세스를 대신해서 커널이 장치에 접근하게 되는데 커널은 각각의 인터페이스를 통해 장치에 접근하게 된다.

리눅스에서 각각의 장치들은 디바이스 파일(device file) 로 관리되고 이 파일을 조작하므로서 장치에 접근할 수 있게 된다.

프로세스가 디바이스 파일을 조작하면 커널 내부의 디바이스 드라이버(device driver) 라고 부르는 소프트웨어가 사용자 대신 장치에 접근하게 된다.

디바이스 드라이버는 장치에 내장된 레지스터 영역을 읽고 쓰면서 요청을 전달하고 장치가 이 요청을 처리하게 된다. 이 요청이 끝나게 되면 디바이스 드라이버가 처리 완료를 확인하게 되는데 이때 폴링(polling) 또는 인터럽트(interrupt) 라는 방식을 사용한다.

- 폴링 (polling) : 채팅 앱에서 주기적으로 앱을 열어 답변이 왔는지 확인하는 방식
- 인터럽트 (interrupt) : 답변이 오면 즉시 앱에서 알림을 보내는 방식

폴링을 하는 동안 CPU는 쉴 수 없기 때문에 인터럽트를 많이 사용하지만 장치의 처리가 빠르고 처리 빈도가 높다면 예외적으로 폴링을 사용하기도 한다.

디바이스 파일에는 다음과 같은 종류가 있는다.

- 캐릭터 장치 : 키보드, 마우스, 단말
- 블록 장치 : HDD, SSD 와 같은 저장 장치
    - SATA나 SAS라면 /dev/sda, /dev/sdb, /dev/sdc가
    - NVMe SSD라면 /dev/nvme1n1, /dev/nvme2n1 과 같은 명으로 디바이스 파일이 생성된다.

우리가 키보드에 입력하면 어떻게 화면에 표시되는 걸까?

  1. 키보드 입력 → 하드웨어 인터럽트 발생.
  2. 커널이 인터럽트를 받아 키보드 디바이스 드라이버 호출.
  3. 디바이스 드라이버가 키보드 신호를 해석해 커널에 전달.
  4. 커널이 변환된 데이터를 적절한 프로세스에 전달.
  5. 해당 프로세스가 입력을 받아 화면에 표시.
  6. 디스플레이 드라이버가 커널로부터 데이터를 받아 화면에 출력.
